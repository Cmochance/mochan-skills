# 补丁差分 / N-day 分析(防御与研究视角)

通过对比"打补丁前 / 后"的二进制,**理解一个安全补丁到底修了什么** —— 用于防御方评估影响面、写检测规则、排资产优先级、以及安全研究中的根因理解。本文保持在方法论 / 工具对比 / 根因定位层面,不提供武器化利用链。

> 前提:只对自有 / 授权 / 公开可获取的更新包做分析;研究遵循负责任披露。

## 什么时候进本域

- 「这个补丁/更新修了什么」「patch diff」「N-day / 1day 分析」
- 「PDB 缺失 / 老版本没符号,想把新版本符号迁移过去」
- 「bindiff / Diaphora / ghidriff 怎么选」
- 「从补丁定位被改的函数 / 根因区域」「写检测规则前先搞清改了哪」

## 总体流程

1. **取样**:拿到补丁前后两个版本的同一二进制(主程序 / DLL / so / 驱动)。来源:官方历史版本、更新包(MSU/MSP 解包、deb/rpm、APK 历史版)。记录版本号、编译时间、架构。
2. **预处理**:两边都先 `rabin2 -I` / 看 header 确认架构、是否带符号、是否加壳。带壳先脱(见 `patterns.md`)。一边有符号一边没有时,先做符号迁移(见下)。
3. **差分**:用二进制 diff 工具做函数级匹配,得到「新增 / 删除 / 改动」函数列表,按"改动小而精"优先看(安全补丁通常是局部加校验 / 改边界 / 补空指针判断)。
4. **定位根因区域**:对改动函数读反编译,对比前后差异 —— 典型修复信号:
   - 新增长度 / 边界 / 范围校验(原来缺 → 溢出 / 越界类)
   - 新增 null / 有效性判断(原来缺 → 空指针 / UAF 类)
   - 整数运算前加溢出检查(整数溢出 → 后续分配/拷贝问题)
   - 类型 / 状态校验、加锁(类型混淆 / 竞态)
   - 输入规范化 / 转义(注入 / 路径穿越类)
5. **理解 + 记录**:用一句话描述"补丁加了什么校验,意味着原来什么输入会出问题",标注函数地址/偏移与置信度。这就是给检测规则 / 影响评估的输入。

## diff 工具对比(按场景选)

| 工具 | 形态 | 适用 | 注意 |
|---|---|---|---|
| **BinDiff** | Ghidra/IDA 配套,GUI | 成熟、匹配质量高,大型 PE/ELF 跨版本对比首选 | 需配合反汇编器;函数匹配靠结构指纹 |
| **Diaphora** | IDA 插件 | 深度,带伪代码 diff、可移植符号/注释 | 依赖 IDA;大库较慢 |
| **ghidriff** | Ghidra headless,命令行 | 自动化/批量、出 Markdown diff 报告,适合 CI / 多版本扫 | 纯命令行,Ghidra 反编译质量边界 |
| **radiff2**(radare2) | CLI | 轻量、快速看字节/函数级差异、脚本化 | 语义匹配弱于上面三者 |

选择口诀:**手动深挖单点 → Diaphora/BinDiff;批量/自动出报告 → ghidriff;快速粗看 → radiff2。**

## 跨版本符号迁移(PDB 缺失 / 老版本没符号)

目的:有符号的一个版本(如新版带调试符号、或开源对应版)→ 把函数名/结构迁到没符号的版本,省去重命名。

- **工具法**:BinDiff / Diaphora 的 "import symbols / port names" —— 按函数匹配置信度迁移名字与注释,低置信度的人工复核。
- **结构指纹**:对匹配靠常量、字符串引用、调用图拓扑、switch 跳表等稳定特征,别只靠地址(地址跨版本必变)。
- **LLM 辅助批量**:对一批未命名函数,把反编译伪代码 + 邻近已知符号上下文喂给模型,让它**提议**函数名/作用,人工抽检确认 —— 适合大量样板函数的初筛,**不可全自动信任**,关键函数必须人工核。`skills/binary-diff/references/prompt-template.md` 有该思路的提示词骨架可参考。

## 纪律

- 结论标注:函数地址/偏移 + "改了什么校验" + 置信度(高/中/低)。
- 区分「确定是安全修复」与「顺带重构/优化」—— 不是每个 diff 都对应漏洞。
- 不确定根因就说不确定,别脑补利用路径。
- 影响评估只到"这类输入会触发问题 + 受影响版本范围",检测规则编写交给 `malware-supplychain.md` 的 YARA/Sigma 部分。
